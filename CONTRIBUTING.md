# Mitwirken am BitcoinPrice Plugin

Vielen Dank für dein Interesse an der Verbesserung des BitcoinPrice Plugins! Hier sind einige Richtlinien, die dir helfen sollen, zum Projekt beizutragen.

## Wie kann ich beitragen?

### Fehler melden

Wenn du einen Fehler findest, erstelle bitte ein Issue im [Issue-Tracker](https://github.com/MartinGeltl21/BitcoinPrice/issues) mit folgenden Informationen:

1. Eine klare und beschreibende Überschrift
2. Eine detaillierte Beschreibung des Fehlers
3. Schritte zum Reproduzieren des Fehlers
4. Die erwartete Funktionalität
5. Das tatsächliche Verhalten
6. Wenn möglich, Screenshots oder Logs

### Neue Funktionen vorschlagen

Vorschläge für neue Funktionen sind immer willkommen! Erstelle ein Issue und beschreibe:

1. Was deine Idee ist
2. Warum sie nützlich für das Plugin wäre
3. Wenn möglich, wie sie implementiert werden könnte

### Code beitragen

1. Forke das Repository
2. Erstelle einen Feature-Branch (`git checkout -b feature/DeineFeature`)
3. Implementiere deine Änderungen
4. Teste deine Änderungen gründlich
5. Committe deine Änderungen (`git commit -m 'Füge eine neue Funktion hinzu'`)
6. Pushe zu deinem Branch (`git push origin feature/DeineFeature`)
7. Eröffne einen Pull Request

## Coding-Richtlinien

### Java-Code-Stil

- Verwende 4 Leerzeichen für die Einrückung, keine Tabs
- Halte dich an die Google Java Style Guide soweit möglich
- Alle öffentlichen Methoden sollten mit JavaDoc-Kommentaren dokumentiert sein
- Verwende sinnvolle Variablen- und Methodennamen

### Commit-Nachrichten

- Verwende klare und beschreibende Commit-Nachrichten
- Beginne mit einem Verb im Imperativ: "Füge hinzu", "Korrigiere", "Entferne", etc.
- Halte die erste Zeile unter 72 Zeichen
- Wenn nötig, füge nach einer Leerzeile einen detaillierteren Beschreibungstext hinzu

## Versionierung

Wir verwenden [Semantic Versioning](https://semver.org/) für die Versionierung dieses Projekts. Das heißt:

- MAJOR-Version für inkompatible API-Änderungen
- MINOR-Version für rückwärtskompatible Funktionalitätserweiterungen
- PATCH-Version für rückwärtskompatible Bugfixes

## Pull Request-Prozess

1. Stelle sicher, dass dein Code gut getestet ist
2. Aktualisiere README.md oder andere Dokumentation, wenn nötig
3. Der Pull Request wird überprüft und bei Bedarf werden Änderungen angefordert
4. Nach der Genehmigung wird der Pull Request gemergt
